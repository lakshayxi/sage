# Pre-Deploy Review: Code + Security

**Date:** 2026-07-18
**Scope:** Full codebase (`/code-review` high-effort, 8-angle multi-agent pass) + a security-focused pass (`/security-review` methodology) against the entire source tree, run before the first public deployment. No prior commits existed, so both reviews treated the whole tree as the diff.

Every finding below was independently verified against the real code (not just asserted) before being reported, and every fix was independently re-verified after landing — tests, ruff, tsc/build, and live curl round-trips for the security-relevant ones. All 10 are fixed.

## Findings and fixes

| # | Severity | Finding | Fix |
|---|---|---|---|
| 1 | **HIGH (security)** | Conversation history had no owner concept at all — `GET /conversations` listed every conversation ever created; `GET /conversations/{id}` returned any transcript by guessable sequential ID. On a public multi-visitor deployment, any visitor could read every other visitor's questions and answers. | Added a server-issued, unguessable `session_token` (`secrets.token_urlsafe(32)`) per conversation. List/get/chat routes now require it (`X-Session-Token` header, or `session_token` query param for `/chat/stream` since `EventSource` can't set headers) and 404/empty-list on mismatch. Frontend persists the token in `localStorage`. **Live-verified**: two independently created sessions can only read their own conversations; cross-session reads 404. |
| 2 | MEDIUM (security) | `DemoKeyMiddleware`'s only mechanism was an `X-Demo-Key` header, but the frontend's `/chat/stream` uses native `EventSource`, which cannot set custom headers — the guardrail couldn't actually be enabled without breaking the app. | Middleware now also accepts the key via a `key` query param; frontend attaches it there for the stream and as a header everywhere else, gated behind a `VITE_DEMO_ACCESS_KEY` build-time var. **Live-verified** both paths against a running server with `DEMO_ACCESS_KEY` set. |
| 3 | HIGH (correctness) | A malformed LLM citation entry missing `"n"` produced `Citation(n=None, ...)`, which crashed `CitationOut` construction outside any try/except — an unhandled 500 in `/chat`, and in `/chat/stream` a successful generation (already streamed and persisted) looked like a failure with no `done` event ever sent. | Malformed entries are now dropped at parse time (logged, not raised); both route handlers wrap response-building so any future mismatch degrades to the existing graceful error instead of crashing. Regression tests added. |
| 4 | HIGH (correctness) | Multi-company retrieval's `_merge_balanced` capped the *merged* candidate pool to one shared `top_k` total, not `top_k` per company — undermining the just-shipped Compare Mode fix, which assumed (per its own comment) that each company already got its own full budget. | `retrieve_hybrid`'s multi-company branch no longer caps the combined result; each company's full candidate list survives to the per-company reranking stage. Regression test asserts per-company, not shared, budget. |
| 5 | MEDIUM (correctness) | The multi-company reranking branch silently dropped any chunk with no company tag (`company=None`/`""`) — reachable in practice via the document-upload form's optional override, which only skips `None`, not empty string. | Untagged candidates now rerank as their own group instead of being excluded. |
| 6 | MEDIUM (convention) | Ingestion stored full chunk text in Chroma's `documents` field, violating this project's own documented rule that SQLite is the sole source of truth for chunk text — and the duplicated text was never even read back. | `pipeline.py` no longer passes chunk text to Chroma; only ids, embeddings, and filter metadata. |
| 7 | MEDIUM (correctness) | The reranker and local embedder's lazy-singleton model loading had no lock; concurrent cold-start requests could each see `None` and both construct a full model. | Added double-checked locking to both, plus model warmup in `lifespan()` at startup so the race can't happen under real traffic. **Live-verified** via startup logs (models load before "Application startup complete"). |
| 8 | LOW-MEDIUM (correctness) | Citation click-to-scroll bypassed the shared highlight context and used `document.getElementById` against an unenforced, hardcoded ID-string convention shared with a sibling component. | Click-to-scroll now goes through `CitationHighlightContext` (a `Map` of registered card refs), same mechanism as hover/focus highlighting. |
| 9 | LOW (correctness) | `POST /documents/upload` had no rate limit, unlike `/chat`, despite doing real per-request PDF-parsing + embedding work. | Added the same `@limiter.limit` decorator. Low real-world severity since `ALLOW_UPLOADS=false` in the actual public deployment. |
| 10 | LOW (correctness) | A wrong-HTTP-method request to a real API path (e.g. `GET /chat`) returned a misleading static-file 404 instead of 405 — a Starlette routing quirk where a root `Mount` always wins over a method-mismatched route. | FastAPI 0.139.2 ships `app.frontend(...)`, a purpose-built low-priority static route checked only after normal routing (including 405s) is exhausted — replaced the plain `StaticFiles` mount with it. **Live-verified**: `GET /chat` now correctly returns 405. |

## Why this is logged separately from `user-testing.md`

`user-testing.md` records bugs found by hand, driving the live app. This was a different kind of pass — a structured, multi-agent adversarial review of the whole codebase before its first public exposure — so it gets its own record rather than being folded into either the user-testing log or a bare commit message.
